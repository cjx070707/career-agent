# 智能求职辅导 Agent PRD

## 1. 项目概述
智能求职辅导 Agent 面向高校学生求职与实习场景，提供岗位匹配、简历优化、投递追踪、面试复盘与个性化辅导能力。项目以大模型、私域数据检索、工具调用与记忆系统为基础，围绕“学生个人求职数据 + 岗位 JD + 历史投递与面试反馈”构建一个可持续陪跑的智能 Agent。

该项目可包装为老师与外部职业教育机构/就业服务机构的合作项目，用于提升学生求职辅导效率、标准化职业咨询流程，并降低重复性辅导成本。

## 2. 项目目标

### 2.1 业务目标
1. 为学生提供 7x24 小时的智能求职辅导服务。
2. 提升岗位匹配效率，减少学生在岗位筛选和信息整理上的时间成本。
3. 帮助学生基于简历、项目经历与历史反馈进行持续优化。
4. 通过智能问答与工具调度，提高求职辅导的标准化和可复用性。
5. 支持老师或合作机构以更低成本服务更多学生。

### 2.2 产品目标
1. 打通候选人信息、简历、岗位 JD、投递记录、面试反馈等核心私域数据。
2. 建立短期会话记忆与长期职业画像体系。
3. 支持连续多轮对话与个性化建议生成。
4. 提供文本问答为主、图片输入与语音播报为辅的多模态体验。
5. 形成一个可复用、可扩展的求职辅导 Agent 应用骨架。

## 3. 产品定位
本项目不是单纯的聊天机器人，而是一个面向学生求职场景的智能 Agent。它的核心能力包括：

1. 基于候选人私域数据和岗位数据进行上下文理解与问答。
2. 调用内部工具完成候选人档案查询、岗位检索、投递记录调取、面试反馈分析等任务。
3. 结合短期记忆和长期职业画像，实现持续、个性化的求职辅导。
4. 支持从历史记录中提炼关键事件与能力短板，用于后续岗位推荐和辅导建议。

## 4. 用户角色

### 4.1 主要用户
1. 在校学生
2. 应届求职者
3. 实习求职者

### 4.2 次要用户
1. 就业指导老师
2. 外部职业教育机构顾问
3. 校企合作项目负责人

## 5. 典型使用场景
1. 学生上传简历，询问“我适合投哪些后端岗位？”
2. 学生提供目标 JD，询问“我的简历该怎么改更匹配？”
3. 学生查看历史投递记录，询问“为什么最近几次都没有进入面试？”
4. 学生在面试结束后录入反馈，询问“我最近面试暴露的共性问题是什么？”
5. 学生持续与 Agent 对话，获取长期职业发展建议和下一步行动计划。

3. 简历与项目经历管理

8. 多模态交互
9. 配置化与模块化扩展

## 8. 详细功能需求

### 8.1 智能问答中心
支持用户通过自然语言提问，系统识别问题意图后，结合候选人档案、简历、岗位 JD、投递记录、面试反馈和记忆上下文生成答案。

典型问题包括：
- 我适合投哪些岗位？
- 这份 JD 和我的经历匹配度怎么样？
- 我的简历应该怎么改？
- 最近几次面试暴露了哪些问题？
- 下一步我该优先做什么？

### 8.2 候选人信息管理
管理候选人的基础信息与求职偏好，包括：
- 姓名、学校、专业、年级
- 技能标签
- 目标岗位方向
- 求职城市偏好
- 薪资期望
- 求职阶段状态

### 8.3 简历与项目经历管理
支持：
- 多份简历录入
- 项目经历录入
- 技术栈标签化
- 项目亮点结构化
- 与目标岗位进行内容匹配

### 8.4 岗位 JD 管理
支持：
- JD 文本录入
- 岗位要求结构化
- 技术关键词提取
- 向量化索引
- 岗位查询与匹配分析

### 8.5 投递与面试记录管理
支持：
- 公司、岗位、投递日期、状态管理
- 面试轮次与面试时间记录
- 面试反馈录入
- 拒绝原因/复盘结论沉淀

### 8.6 记忆系统
建立短期记忆与长期记忆：

#### 短期记忆
保存最近几轮用户与 Agent 的对话内容，用于连续问答上下文继承。

#### 长期记忆
沉淀：
- 长期职业画像
- 目标岗位偏好
- 技能强弱项
- 常见面试问题
- 历史关键事件

### 8.7 Agent 工具调度
将内部能力封装为工具，例如：
- 查询候选人档案
- 查询简历
- 查询项目经历
- 查询岗位 JD
- 查询投递记录
- 查询面试反馈
- 分析岗位匹配度

由 Agent 根据用户问题自动选择工具并整合结果。

### 8.8 多模态交互
第一版支持：
- 上传简历截图
- 上传岗位截图
- 可选语音播报输出

## 10. Agent 主链路

### 10.1 核心流程
1. 接收用户问题
2. 尝试识别当前候选人上下文
3. 读取短期记忆、长期画像、关键事件
4. 将记忆上下文注入 Agent
5. Agent 根据问题生成规划并调用工具
6. 汇总结构化结果并生成最终答案
7. 回写短期记忆
8. 达到阈值后自动沉淀长期画像与关键事件

### 10.2 核心设计原则
1. 不直接只靠 prompt 回答
2. 优先通过工具获取结构化证据
3. 回答基于候选人私域数据
4. 记忆系统形成长期陪跑闭环

## 11. 检索设计

### 11.1 检索对象
1. 岗位 JD
2. 项目经历
3. 面试反馈
4. 投递记录
5. 长期关键事件

### 11.2 检索方式
1. 关键词检索
2. 向量检索
3. 混合召回与排序

### 11.3 检索目标
支持用户提出自然语言问题时，系统召回最相关的：
- 岗位要求
- 历史失败模式
- 可复用项目亮点
- 阶段性建议依据

## 12. 页面与交互要求

### 12.1 Query 页面
支持输入问题、上传图片、获取回答，用于单次任务型问答。

### 12.2 Chat 页面
支持多轮对话，展示上下文与持续辅导过程。

### 12.3 数据管理页面
至少支持：
- 候选人信息查看
- 岗位 JD 查看
- 投递/面试记录查看

## 13. 技术方案建议
1. 后端：`FastAPI`
2. 数据库：`SQLite`
3. 向量检索：`ChromaDB` 或 `FAISS`
4. 模型：`OpenAI API / 通义千问 / 豆包` 任选一种
5. 前端：`React` 或简单静态页面

## 14. 验收标准
第一版完成后，至少满足：

1. 可以录入候选人、简历、岗位、投递和面试数据。
2. 可以通过 `API + 页面` 完成基础数据查看和管理。
3. Agent 可以根据用户问题自动调用不同工具。
4. 系统支持多轮连续问答。
5. 系统可以把问答回写为短期记忆。
6. 系统可以基于历史记录提炼长期职业画像与关键事件。
7. 系统支持对岗位 JD、项目经历和面试反馈进行检索。
8. 系统至少具备一个可完整演示的场景：
   - 岗位匹配
   - 简历优化
   - 面试复盘
   - 下一步行动建议

## 15. 一句话总结
这是一个面向学生求职场景的多模态智能 Agent 项目，核心是把候选人私域数据、岗位 JD、投递与面试记录、记忆系统和工具调用整合进一条可持续演进的求职辅导链路中，形成一个具备真实业务感和较强工程表达能力的 AI 应用项目。
